## 2026-03-29 — First officer perspective on operating in a Spacedock workflow

**Project:** Spacedock docs/plans workflow — self-dogfood, managing Spacedock's own development.
**Usage level:** Heavy. 8 active tasks, up to 3 concurrent agents, worktree isolation, PR lifecycle via pr-merge mod.

### What works well

**State continuity across sessions.** The entity files are genuinely useful. When I start up, I can read the frontmatter and stage reports to understand where everything is. Without this, I'd be starting cold every session — the captain would have to re-explain what's in flight. The YAML frontmatter is the right level of structure: enough to parse programmatically (status script), simple enough that I can edit it with a single `Edit` call.

**Stage definitions as guardrails.** The Inputs/Outputs/Good/Bad structure for each stage keeps agents focused. Without it, an ensign asked to "validate" would invent its own criteria. The stage definition channels effort — the validator knows to check the log against findings, not rubber-stamp. Same for ideation: "produce acceptance criteria" is a concrete deliverable, not "think about stuff."

**Worktree isolation.** This is the single biggest win. Without it, concurrent agents would step on each other's files constantly. The worktree model gives each agent a clean checkout to work in. Merge conflicts are rare because the work is naturally scoped.

**The mod system (pr-merge).** Cleanly separates "how the workflow works" from "what happens at boundaries." The PR lifecycle is complex (push, create PR, detect merge, archive) but it's all in one mod file. The FO doesn't hardcode any of it.

### What feels like too much constraint

**The validation stage on template-only changes.** For task 072 (adding a "Working Directory" section to the FO template), the implementation is 6 lines of prose. The validation ensign will spend more tokens reading the file, running the test harness, and writing a stage report than the entire change is worth. A simple diff review by the captain would be faster and higher-signal. The workflow doesn't distinguish between "big risky change" and "trivial wording fix."

**Gate ceremony for backlog → ideation.** Every task sits in backlog until the captain explicitly approves it for ideation. For tasks the captain just filed themselves (like 072, which they told me to file), requiring a separate "approve" to move it feels like a round-trip that adds no information. The captain already decided they want this done.

**Stage reports as structured checklists.** The checklist format (- [x] item / evidence) works well for implementation and validation where there are concrete deliverables. For ideation, it feels forced — the real output is the approach and acceptance criteria in the body, not a meta-report about having written them. The stage report ends up restating what's already in the sections above it.

### What I'm unsure about

**Feedback rejection cycles.** The 033 experiment got rejected by the validator with suggestions to retest. The mechanism works (validator identifies issues, FO presents at gate, captain decides). But I'm uncertain whether the captain wants to spend tokens on a redo here, or whether the finding (agent types frozen at init) is sufficient to close the task. The workflow doesn't have a "close with partial findings" path — it's approve/reject binary.

**Concurrency limits.** The README says `concurrency: 2` but I've been running 3 agents at once (033 implementation + 072 ideation + holding 033 validator at gate). The concurrency limit feels like it should be about worktree stages specifically (where file conflicts matter), not about all agents. An ideation ensign working on main can't conflict with a worktree agent.

**How much the FO should editorialize at gates.** I've been adding "Recommend approve" to gate reviews. But I'm the dispatcher, not the reviewer — my opinion on whether ideation is good enough is less informed than the captain's. Not sure if my recommendations add value or just add noise.

### Compared to working without a workflow

Without Spacedock, I'd be a single agent doing everything sequentially in one context. The captain would say "implement X" and I'd ideate, implement, and self-validate in one shot. That's faster for small tasks but falls apart for anything that needs multiple sessions — I'd lose context between conversations and the captain would have to re-orient me every time.

The workflow adds overhead per task but makes multi-session, multi-agent work tractable. The entity file is the shared memory I don't otherwise have. The stage definitions prevent the "agent does whatever it feels like" failure mode. The trade-off is real: simple tasks feel over-processed, complex tasks benefit significantly.

### Specific friction (bugs/issues observed this session)

1. **Agent cwd drift** — FO `cd`'d into a worktree, spawned agent inherited wrong cwd (task 072 filed)
2. **CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=0 has no effect** — env var doesn't disable team tools
3. **Agent types frozen at session init** — dynamically-created agents unavailable mid-session
4. **Model self-censoring** — mission description biased experiment behavior
5. **Validation scope gap** — inline pilot vs fresh session distinction missed in experiment design
